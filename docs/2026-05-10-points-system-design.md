# 留情局积分与等级系统设计文档

**日期**: 2026-05-10
**版本**: v1.2

---

## 1. 系统概述

留情局将引入**积分系统**作为核心激励机制，结合**等级系统**来激励用户参与社区活动。

### 1.1 核心概念

- **积分（余额）**：用户当前持有的积分数量，可增可减，用于消费和交易
- **贡献值**：用户累计获得的积分总和，只增不减，用于等级评判
- **等级**：基于贡献值划分的用户荣誉称号

### 1.2 核心规则

```
积分余额 = 获得积分 - 消耗积分（可增可减）
贡献值 = 累计获得的积分（只增不减）
等级 = 根据贡献值计算
```

### 1.3 运营消耗机制

- 资源下载：下载者消耗10积分 → 发布者获得7积分（扣除30%运营费）
- 平台留存30%用于社区运营和发展

## 2. 积分系统

### 2.1 积分获取规则

| 行为 | 获得积分 | 余额变化 | 贡献值变化 |
|------|---------|---------|-----------|
| 每日首次登录 | +5 | +5 | +5 |
| 发布论坛帖子 | +10 | +10 | +10 |
| 回复帖子 | +3 | +3 | +3 |
| 点赞帖子 | +1 | +1 | +1 |
| 发布资源 | +15 | +15 | +15 |
| 下载资源（发布者获得） | +7 | +7 | +7 |
| 完成悬赏（完成者获得） | 悬赏金额 | +悬赏金额 | +悬赏金额 |

### 2.2 积分消耗规则

| 行为 | 消耗积分 | 余额变化 | 贡献值变化 |
|------|---------|---------|-----------|
| 发布悬赏 | 悬赏金额 | -悬赏金额 | 不变 |
| 下载资源 | 10 | -10 | 不变 |

### 2.3 积分收支平衡

```
贡献值 = 余额 + 已消耗积分总额

示例：
用户获得 100 积分（贡献值 = 100）
- 下载资源消耗 100 → 余额 0，贡献值 100
- 下载资源消耗 30 → 余额 0（需要先获得积分），贡献值 100
- 发悬赏消耗 30 → 余额 20，贡献值 100
- 完成悬赏获得 50 → 余额 70，贡献值 150
```

### 2.4 示例说明

```
用户A（下载者）的操作流程：
1. 第一次登录 → 余额 5，贡献值 5
2. 发帖 +10 → 余额 15，贡献值 15（更新）
3. 发资源 +15 → 余额 30，贡献值 30（更新）
4. 发悬赏消耗 -10 → 余额 20，贡献值 30（不变）
5. 完成别人悬赏 +20 → 余额 40，贡献值 50（更新）
6. 下载别人资源 -10 → 余额 30，贡献值 50（不变）
7. 再次下载别人资源 -10 → 余额 20，贡献值 50（不变）

资源发布者B的操作流程（同一资源被下载3次）：
8. B的资源被下载 → B获得 +7，贡献值更新
9. B的资源被下载 → B获得 +7，贡献值更新
10. B的资源被下载 → B获得 +7，贡献值更新
   （平台每次获得3积分，共获得9积分作为运营费）
```

## 5. 技术实现

### 5.1 PointsService 核心方法

```javascript
const PointsService = {
  // 积分操作
  addTokens(userId, amount, type, description, relatedId),
  deductTokens(userId, amount, type, description, relatedId),

  // 查询
  getBalance(userId),           // 获取当前余额
  getContributionValue(userId), // 获取贡献值
  getLevel(userId),            // 获取等级信息
  getHistory(userId, limit),    // 获取历史记录

  // 每日登录检查
  checkDailyLogin(userId),

  // 等级计算
  calculateLevel(contributionValue)
}
```

### 5.2 关键逻辑

**addTokens() 方法**:
- 增加 tokens 余额
- 增加 contributionValue（用于等级）
- 增加 totalEarnings
- 记录历史

**deductTokens() 方法**:
- 减少 tokens 余额
- **不**减少 contributionValue
- 记录历史

## 7. 注意事项

1. **防止刷分**：每日登录奖励每天只生效一次
2. **数据一致性**：addTokens 同时更新余额和贡献值，deductTokens 只更新余额
3. **错误处理**：积分不足时阻止操作
4. **历史记录**：保留积分变动记录便于追溯
